### 2. `SkillManager` 类实现

#### 核心特性
- **线程安全**：使用 `threading.Lock` 确保文件读写安全
- **容错处理**：处理空文件、损坏的 JSON、缺失字段等异常情况
- **持久化存储**：技能库存储在 `browser_data/skills_library.json`
- **纯粹的经验管理类**：不参与网页执行，不调用其他模块

#### 初始化与文件管理

**`__init__(skills_file_path)`**
- 默认路径：`browser_data/skills_library.json`
- 自动创建目录（如果不存在）
- 初始化时加载技能库
- 线程锁保护文件操作

**`_load_skills()`**
- 从 JSON 文件加载技能库
- 容错处理：
  - 文件不存在 → 创建空文件
  - 文件为空 → 返回空列表
  - JSON 解析失败 → 重置为空列表
  - 格式错误（非列表）→ 重置为空列表

**`_save_skills(skills)`**
- 线程安全的文件写入
- 使用 `ensure_ascii=False` 支持中文
- 使用 `indent=2` 格式化输出

#### 反思提炼方法

**`async def reflect_and_extract(goal, action_history, llm_client)`**
- **入参**：
  - `goal`: 任务目标
  - `action_history`: 操作历史列表
  - `llm_client`: LLMClient 实例
- **逻辑**：
  1. 格式化反思输入（过滤废动作、标记死循环警告）
  2. 调用 LLM 进行反思（使用专属的 Reflection System Prompt）
  3. 提取 SOP 文本
- **返回**：提炼后的通用 SOP 字符串

**反思 Prompt 核心约束**：
```
1. 去具体化：禁止保留 element_id、选择器、坐标
2. 过滤废动作：过滤试错、死循环警告、兜底动作
3. 提炼主线：只保留核心主线逻辑
4. 语义描述：使用自然语言而非技术细节
5. 通用性：适用于类似任务场景
```

**输入格式化示例**：
```
【任务目标】
在淘宝搜索商品

【操作历史】（共 6 步）
1. navigate url='https://www.taobao.com'
   思考: 打开淘宝首页
2. click [element_id=5]
   思考: 点击搜索框
3. type [element_id=5] text='iPhone 15'
   思考: 输入商品名称
4. [废动作] system_warning: 死循环熔断
5. click [element_id=6]
   思考: 点击搜索按钮
6. done
   思考: 任务完成

【任务要求】
请提炼出达成目标的核心主线逻辑，输出为精简的通用步骤 SOP。
记住：绝对禁止保留具体的 element_id，必须转为语义描述。
```

#### 保存新技能方法

**`save_new_skill(goal, sop_text)`**
- **入参**：
  - `goal`: 任务目标（作为 `intent_description`）
  - `sop_text`: 提炼后的 SOP（作为 `guidance_sop`）
- **逻辑**：
  1. 生成全局唯一的 `skill_id`（使用 `uuid.uuid4().hex`）
  2. 组装技能数据结构（符合 CONTRACT.md 规范）
  3. 追加到技能列表
  4. 持久化到 JSON 文件
- **返回**：新创建的技能字典

**技能数据结构**（符合 CONTRACT.md）：
```json
{
  "skill_id": "428f7d7e5dc940a6a7dbb8056c59e0a9",
  "intent_description": "在淘宝搜索商品",
  "guidance_sop": "1. 导航到淘宝首页\n2. 在搜索框输入商品名称\n3. 点击搜索按钮",
  "created_at": "2026-04-30T07:11:17.307135Z",
  "success_count": 0
}
```

#### 技能检索方法

**`search_skills_by_intent(query, top_k=3)`**
- **简单版本**：基于关键词匹配
- **匹配策略**：
  - 在 `intent_description` 中匹配（权重 2）
  - 在 `guidance_sop` 中匹配（权重 1）
  - 按分数排序，返回前 k 个
- **未来扩展**：
  - 使用向量数据库进行语义检索
  - 使用 LLM 进行意图匹配

#### 辅助方法

**`get_all_skills()`**
- 获取所有技能（返回副本）

**`get_skill_by_id(skill_id)`**
- 根据 skill_id 获取技能
- 不存在则返回 None

**`increment_success_count(skill_id)`**
- 增加技能的成功复用次数
- 自动持久化到文件
- 返回是否成功更新

## 核心设计决策

### 存储路径选择
- **默认路径**：`browser_data/skills_library.json`
- **理由**：
  - `browser_data` 目录已存在，用于存储浏览器相关数据
  - 与截图、缓存等数据放在一起，便于管理
  - 可以通过参数自定义路径

### 线程安全设计
- **使用 `threading.Lock`**：确保多线程环境下的文件读写安全
- **锁的粒度**：仅在文件操作时加锁，不影响其他操作

### 反思 Prompt 设计
- **核心约束明确**：5 条核心约束，确保 LLM 输出符合要求
- **示例驱动**：提供正确和错误的示例，帮助 LLM 理解
- **格式化输入**：清晰的结构化输入，便于 LLM 理解

### 简单检索策略
- **当前版本**：基于关键词匹配
- **优点**：
  - 无外部依赖
  - 实现简单
  - 性能高
- **缺点**：
  - 语义理解能力弱
  - 无法处理同义词
- **未来扩展**：
  - 使用向量数据库（如 Chroma、FAISS）
  - 使用 LLM 进行意图匹配

## 性能指标

### 文件操作
- **读取**：O(1)，直接读取 JSON 文件
- **写入**：O(n)，序列化整个技能列表
- **线程安全**：使用锁保护，无竞态条件

### 检索性能
- **时间复杂度**：O(n*m)，n 为技能数量，m 为查询词数量
- **空间复杂度**：O(n)，存储所有技能
- **优化空间**：使用倒排索引可优化到 O(m)

### 存储空间
- **单个技能**：约 200-500 字节（取决于 SOP 长度）
- **1000 个技能**：约 200-500 KB
- **可扩展性**：JSON 文件适合小规模（< 10000 个技能）
